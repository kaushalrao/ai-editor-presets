# Python Rules

## Style & Formatting

- Follow PEP 8. Assume `ruff` is configured — do not fight it's formatting.
- Max line length: 88 (black default). Never argue for 79.
- Use double quotes for strings (black default). Single quotes only inside f-strings to avoid escaping.
- Imports order: stdlib → third-party → local. One blank line between groups. Use `isort` conventions.
- Never use wildcard imports (`from module import *`). Always explicit.
- Prefer `pathlib.Path` over `os.path` for all filesystem operations.

## Type Annotations

- All function signatures must have type annotations — parameters and return type.
- Use `from __future__ import annotations` at top of file for forward references (Python < 3.12).
- Prefer built-in generics over `typing` equivalents where available (3.9+):
  - `list[str]` not `List[str]`
  - `dict[str, int]` not `Dict[str, int]`
  - `tuple[int, ...]` not `Tuple[int, ...]`
  - `X | None` not `Optional[X]`
  - `X | Y` not `Union[X, Y]`
- Use `TypeAlias` for complex type aliases. Use `TypeVar` only when genuinely generic.
- Use `Protocol` for structural subtyping over abstract base classes where possible.
- Use `TypedDict` for dict shapes passed across boundaries (e.g., JSON payloads, config dicts).
- `Any` is a last resort. Document with a comment why it's unavoidable.

```python
# Good
def process(items: list[str], limit: int | None = None) -> dict[str, int]:
    ...

# Bad
from typing import List, Optional, Dict
def process(items: List[str], limit: Optional[int] = None) -> Dict[str, int]:
    ...
```

## Naming Conventions

- `snake_case` for variables, functions, modules.
- `PascalCase` for classes.
- `SCREAMING_SNAKE_CASE` for module-level constants.
- `_single_leading_underscore` for internal/private (not enforced by Python, but communicates intent).
- `__dunder__` only for Python protocol methods — never invent new dunder names.
- Avoid abbreviations unless universally understood (`cfg`, `ctx`, `db`, `req`, `res` are fine).
- Boolean variables/functions should read as predicates: `is_valid`, `has_permission`, `can_retry`.

## Functions & Methods

- Functions do one thing. If you need "and" to describe it, split it.
- Max ~20 lines per function body. Beyond that, extract helpers.
- Prefer keyword arguments for functions with more than 2 parameters.
- Use `*` to enforce keyword-only args for readability at call sites:

  ```python
  def send_email(*, to: str, subject: str, body: str) -> None: ...
  ```

- Never use mutable defaults. Use `None` sentinel + guard:

  ```python
  def append(items: list[str] | None = None) -> list[str]:
      if items is None:
          items = []
      return items
  ```

- Prefer returning early over deeply nested conditionals (guard clauses).
- Use `@functools.cache` or `@functools.lru_cache` for pure functions with repeated calls.

## Classes

- Use `@dataclass` or `pydantic.BaseModel` over manual `__init__` for data-holding classes.
- Prefer `@dataclass(frozen=True)` for immutable value objects.
- `__repr__` should be auto-generated by dataclass or explicitly defined — never rely on default.
- Use `__slots__` on hot-path classes to reduce memory overhead.
- Avoid multiple inheritance. Prefer composition. Mixins acceptable only when thin and single-purpose.
- `@classmethod` for alternate constructors. `@staticmethod` only if it truly doesn't need `cls`/`self`.
- `@property` for computed attributes; avoid it when computation is non-trivial (surprising to callers).

## Error Handling

- Catch specific exceptions — never bare `except:` or `except Exception:` unless re-raising.
- Define custom exceptions inheriting from a project-level base exception:

  ```python
  class AppError(Exception): ...
  class ValidationError(AppError): ...
  class NotFoundError(AppError): ...
  ```

- Always include context when re-raising: `raise NewError("ctx") from original_exc`.
- Use `contextlib.suppress(ExcType)` for intentional swallowed exceptions — never silent `except: pass`.
- Do not use exceptions for control flow (e.g., don't raise to signal "empty result").

## Data Validation

- Use `pydantic` v2 for all external data (API payloads, config, env vars).
- Use `pydantic-settings` for environment-based config with `.env` support.
- Validate at the boundary (entry point). Assume internal data is already valid — don't re-validate downstream.
- Use `Annotated` + `Field(...)` for constraints:

  ```python
  from typing import Annotated
  from pydantic import Field
  PositiveInt = Annotated[int, Field(gt=0)]
  ```

## Async

- Use `async`/`await` consistently. Never mix sync blocking I/O into async code paths.
- Use `asyncio.TaskGroup` (3.11+) for concurrent tasks instead of `gather` where possible.
- Use `anyio` for library code that should be backend-agnostic (asyncio/trio).
- Annotate coroutines with return type: `async def fetch() -> bytes: ...`
- Use `asyncio.run()` only at the top-level entry point.
- Avoid `asyncio.sleep(0)` hacks — restructure the logic instead.

## Logging

- Use stdlib `logging`, never `print()` in production code.
- Get logger per module: `logger = logging.getLogger(__name__)`.
- Use `structlog` for structured/JSON logging in services.
- Log at appropriate levels: `DEBUG` for tracing, `INFO` for lifecycle, `WARNING` for recoverable issues, `ERROR` for failures, `CRITICAL` only for unrecoverable states.
- Never log sensitive data (tokens, passwords, PII). Mask or omit.
- Prefer `logger.exception("msg")` over `logger.error("msg", exc_info=True)` inside except blocks.

## Testing

- Use `pytest`. No `unittest.TestCase` unless integrating legacy code.
- Test file mirrors source: `src/auth/service.py` → `tests/auth/test_service.py`.
- One logical assertion per test (multiple `assert` lines are fine if testing one behaviour).
- Name tests: `test_<unit>_<scenario>_<expected_outcome>`.
- Use `pytest.fixture` with appropriate scope (`function` default, `session` for expensive setup).
- Use `pytest-mock` (`mocker` fixture) over `unittest.mock` directly.
- Mark slow/integration/e2e tests: `@pytest.mark.integration` — keep unit tests fast (<1s total).
- Aim for tests on behaviour, not implementation. Avoid testing private methods.
- Use `hypothesis` for property-based testing on pure functions with complex input spaces.

```python
# Good test name
def test_transfer_funds_insufficient_balance_raises_error(): ...

# Bad
def test_transfer(): ...
```

## Project Structure

```
project/
├── src/
│   └── mypackage/         # src-layout (preferred)
│       ├── __init__.py
│       ├── config.py      # pydantic-settings config
│       ├── exceptions.py  # project-wide exception hierarchy
│       ├── models/        # domain models / pydantic schemas
│       ├── services/      # business logic
│       ├── repositories/  # data access layer
│       └── api/           # framework layer (FastAPI routers, etc.)
├── tests/
│   ├── conftest.py
│   ├── unit/
│   └── integration/
├── Pipfile                # single source of truth for deps + tooling
└── Makefile               # common tasks: lint, test, run
```

- Use **src-layout** — prevents accidental imports of non-installed package during tests.
- `Pipfile` only — no `setup.py`, no `setup.cfg` duplication.
- Pin exact versions in `requirements.lock` (or use `poetry.lock` / `uv.lock`), loose ranges in `pyproject.toml`.

## Dependency Management

- Prefer `uv` for speed. `pipenv` acceptable. Avoid raw `pip` in team projects.
- Separate dependency groups: `[project]` deps, `[dev]`, `[test]`, `[docs]`.
- Never commit secrets. Use `python-dotenv` + `.env.example` pattern.

## Performance

- Profile before optimizing. Use `cProfile` + `snakeviz` or `py-spy` for sampling.
- Prefer list/dict/set comprehensions over `map`/`filter` for readability, but not at the cost of clarity.
- Use generators (`yield`) for pipelines over large datasets — avoid loading full datasets into memory.
- Prefer `collections.defaultdict`, `collections.Counter`, `heapq` over re-implementing.
- Use `__slots__` on dataclasses instantiated millions of times.
- For CPU-bound: `multiprocessing` or `concurrent.futures.ProcessPoolExecutor`. For I/O-bound: `asyncio` or `ThreadPoolExecutor`.

## Security

- Never `eval()` or `exec()` on external input.
- Use `secrets` module for tokens, never `random`.
- Sanitize all inputs used in shell commands; prefer `subprocess` with list args (no `shell=True`).
- Use `hashlib` with proper algorithms (`sha256`+). Never MD5/SHA1 for security purposes.
- Validate and sanitize file paths from user input to prevent path traversal.
